## C. How the Theme Wrapper Works

```
npm run build
    |
    v  (vite.config.js sets base = '/wp-content/themes/atoc-bar-theme/')
dist/
  assets/
    index-[hash].js   <- compiled React app
    index-[hash].css  <- compiled styles
    brand/            <- logo
    legacy/           <- venue images
    references/       <- contact sheets
    |
    v  (Build-ATOC-Theme.ps1 copies to stable names)
atoc-bar-theme/
  assets/
    js/app.js         <- deployed to live server
    css/app.css       <- deployed to live server
    brand/
    legacy/
    references/
  functions.php       <- enqueues app.css, injects app.js as <script type="module">
  front-page.php      <- WordPress serves <div id="root"> as front page
    |
    v  (GitHub Actions via FTP)
Live: public_html/wp-content/themes/atoc-bar-theme/
```

**Why `atoc-bar-theme/assets/` is gitignored:**
The compiled JS/CSS and image copies are generated by `Build-ATOC-Theme.ps1`. They are build artifacts, not source files. GitHub Actions runs the build script during CI, which creates them before deployment. This keeps the repo clean.

**Why `import.meta.env.BASE_URL` matters:**
In dev (`npm run dev`), `BASE_URL = '/'` so image paths resolve as `/assets/legacy/front-entrance.jpg`. In the production build, `BASE_URL = '/wp-content/themes/atoc-bar-theme/'` so image paths resolve as `/wp-content/themes/atoc-bar-theme/assets/legacy/front-entrance.jpg`. This is how the same React code works correctly in both the local Vite dev server and WordPress.

## G. WordPress SPA Routing Note

The React app uses `window.location.pathname` for routing. When WordPress serves the site:

- `/` — WordPress uses `front-page.php` (returns HTTP 200, React renders Home) **OK**
- `/about` — WordPress has no "about" page, returns 404, `404.php` loads React, React renders About page content **functional but returns HTTP 404 status**
- Same for `/events`, `/menus`, `/promotions`, `/gallery`, `/bookings`, `/contact`

**For full pre-deployment:** Create a WordPress page for each sub-route (or add a server rewrite rule) so WordPress returns HTTP 200 for those paths. This matters for SEO crawlers. Since the site currently has `noindex,nofollow` set everywhere, the HTTP status issue does not affect live users or crawlers yet. Resolve before enabling indexing.
